## The problem

A trading desk has rules. A medical system has rules. A government API has rules. You can encode those rules in scattered `if`s sprinkled through the codebase, or in a config file that nobody reads, or in a comment block above the function. None of those survive a compliance audit. None of those compose.

## The shape of the code

An `Ethos` is a structured set of obligations and prohibitions. Each rule is named, individually testable, and individually replaceable. The Ethos verifies an action by evaluating every rule and collecting the violations.

```rust
use deep_causality_ethos::*;

let ethos = Ethos::new()
    .forbid("pii_leaves_region",          rule::pii_leaves_region)
    .require("audit_log_present",         rule::audit_log_present)
    .require("approval_when_over_10k",    rule::approval_when_over(10_000))
    .require("kyc_completed_for_user",    rule::kyc_completed_for_user);

let outcome = ethos.verify(&action);

match outcome {
    Verdict::Pass => commit(action)?,
    Verdict::Fail(violations) => {
        for v in violations.iter() {
            audit_log.write(v)?;
        }
        return Err(reject_with(violations));
    }
}
```

## Walkthrough

`forbid` and `require` accept a rule name and a rule closure. Each rule receives the action and returns a boolean. The Ethos evaluates every rule against the action and collects a `Vec<Violation>` for every rule that fails. The verdict is the union of those violations.

The structure that matters is the *name*. Rules carry stable identifiers so a violation is auditable in the operational log, traceable to the rule definition, and quotable in a post-incident review.

The composition is open. New rules land as new `forbid` or `require` calls. Removed rules are removed cleanly. The order does not matter; the Ethos evaluates every rule regardless of which previous rule failed.

## Run it

```bash
cargo new --bin compliance
cd compliance
cargo add deep_causality_ethos
# define your rule closures in src/rule.rs and wire `action` to the request type you care about
cargo run --release
```

## Why this is a good fit

The set of operational rules grows over a product lifetime and a regulatory cycle. Pulling them out of business logic and into an Ethos keeps the rules reviewable and the business code small.
