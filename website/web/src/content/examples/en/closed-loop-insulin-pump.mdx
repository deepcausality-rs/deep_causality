The example lives at [`corrective_glucose_pump`](https://github.com/deepcausality-rs/deep_causality/tree/main/examples/causal_intervention_examples/corrective_glucose_pump). It demonstrates the corrective sense of `intervene`: a closed-loop control action that runs tick by tick, monitors the value channel after each step, and substitutes a corrected value when the carried state drifts outside the safe envelope.

This is a different operational pattern from the [counterfactual examples](/examples/cate-treatment-effect/). Counterfactuals run once, retrospectively, on a static history. Corrective interventions run continuously, prospectively, with each tick deciding whether the next one needs correction. The same `Intervenable::intervene` method serves both; the difference is the loop structure around it.

## The patient model

The chain simulates blood glucose in a type-1 diabetic patient across a six-hour window. Three meal events deliver carbohydrate boluses on a fixed schedule. Hepatic baseline production runs continuously. Without insulin, glucose climbs monotonically and crosses the ketoacidosis line at 300 mg/dL.

The state carries the running glucose, the cumulative bolus count, and the total insulin units delivered. The context (immutable `PumpConfig`) carries the hyperglycemic threshold at 180 mg/dL, the corrective target, the insulin sensitivity factor, and the meal schedule.

## The closed-loop tick

```rust
fn run_closed_loop() -> PumpProcess<FloatType> {
    let cfg = nominal_pump_config();
    CausalFlow::from(model::initial_process())
        .iterate_n(N_TICKS as usize, |tick| {
            tick.bind(model::simulate_step).branch_with(
                |glucose, _state, ctx| {
                    *glucose > ctx.expect("PumpConfig present").hyperglycemic_threshold
                },
                |hot| {
                    hot.update_state(|mut state, &glucose| {
                        let (_, units) = model::corrective_bolus(glucose, &cfg);
                        state.bolus_count += 1;
                        state.total_insulin_units += units;
                        state
                    })
                    .intervene_if(|_| true, |glucose| model::corrective_bolus(glucose, &cfg).0)
                },
                |cold| cold,
            )
        })
        .into_process()
}
```

[`main.rs:74`](https://github.com/deepcausality-rs/deep_causality/blob/main/examples/causal_intervention_examples/corrective_glucose_pump/main.rs#L74). `iterate_n` runs the tick `N_TICKS` times. Each tick advances the chain one step with `bind(simulate_step)`, then `branch_with` reads the post-tick glucose, the state, and the context together. The predicate compares the glucose against the hyperglycemic threshold carried in the context. When it is exceeded, the hot arm runs: `update_state` records the bolus and accumulates the delivered units, and `intervene_if` substitutes the corrected glucose into the value channel. When it is not, the cold arm passes the tick through unchanged. The next iteration advances from whichever value the branch produced.

## What `intervene` adds over a branch in `simulate_step`

A simpler design would put the threshold check inside `simulate_step` itself: if glucose is high, apply a bolus before returning. That works numerically. It loses three things.

First, the separation of concerns. The dynamics (baseline production plus meal absorption) and the control policy (when and how much to dose) become one function. Any change to either requires understanding both.

Second, the audit trail. The closed-loop run's `EffectLog` contains one entry per tick recording the glucose reading, plus one entry per bolus event recording the substitution: `!!Intervention!!: <pre-bolus glucose> replaced with <target>`. A reviewing physician can see exactly which readings triggered which boluses, without instrumenting the dynamics function.

Third, the comparability. The example runs the same chain twice: once with no monitor (open loop) and once with the monitor present (closed loop). The `iterate_n` tick is identical between runs. The only difference is the `branch_with` arm that fires `.intervene`. The open-loop trajectory is the failure the corrective interventions prevent; without the dynamics and control being separated at the chain level, that comparison would require two different functions.

## The trajectories diverge

The open-loop run crosses the ketoacidosis threshold within the six-hour window. The closed-loop run uses the same meal schedule and the same patient parameters; each hyperglycemic excursion triggers a bolus, glucose returns to the safe range, and the patient finishes the window in normoglycemia. The summary prints the bolus count and total insulin units for the closed-loop run, plus the peak glucose for both. The `EffectLog` for the closed-loop run lists every tick reading and every bolus event in order.

## Where this pattern goes

The shape, `iterate_n(n, |tick| tick.bind(step).branch_with(crossed, |hot| hot.intervene_if(..), |cold| cold))`, is the corrective-intervention pattern for any monitored system. Lane-keeping under crosswind, decompression stops during a dive ascent, active/standby network failover. The example crate contains three siblings that apply this pattern to those domains; the [network failover example](/examples/active-standby-failover/) is the most directly comparable. The shape stays the same; only the threshold predicate and the corrective value differ.
